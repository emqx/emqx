Introduced fine-grained scope-based access control for dashboard login users. Each user record now carries an optional `scopes` field; when set, requests are authorised against the same path-to-scope catalogue used for API keys, layered on top of the existing role-based check. Four new scopes (`user_management`, `mfa_management`, `sso_management`, `api_key_management`) cover dashboard-only endpoints and are admin-only except `mfa_management`, which any role may hold for self-exemption from forced MFA. API keys cannot hold any of the four login-only scopes, and the `publisher` API-key role is now constrained to `[publish]` only — both checks apply to the HTTP API and to bootstrap-file loading (incompatible scopes are dropped with a warning).

Two new public catalogue endpoints expose the scope vocabulary for UI consumption: `GET /api_key_scopes` (10 entries) and `GET /user_scopes` (14 entries with an `admin_only` flag). Both are top-level paths so they cannot be intercepted by sibling wildcard routes, and both are accessible to any bearer-authenticated caller.

Added per-SSO-backend `force_mfa` enforcement. Each backend (LDAP, OIDC, SAML) accepts an independent `force_mfa` flag; the value at provision time is snapshotted onto the user record so subsequent backend reconfiguration does not retroactively flip the policy. When the snapshot is true, the user cannot disable or rotate their own MFA without holding `mfa_management` scope, closing the self-bypass window present in earlier releases. Pre-existing SSO users get the field populated lazily on next login.
